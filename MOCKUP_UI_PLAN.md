# Daylog Mock UI Plan

> 기능 구현 전에 화면 흐름과 사용 경험을 먼저 검증하기 위한 mock up UI 기획 문서

---

## 목적

Daylog의 MVP는 일일 운동·식단 기록과 흔들어서 해제하는 미라클 모닝 알람이다.
mock UI 단계에서는 DB, 알림, 센서, 사진 저장 같은 실제 기능보다 사용자가 앱을 어떻게 탐색하고 입력하고 반응하는지를 먼저 확인한다.

이 단계의 목표는 다음과 같다.

- 앱을 켰을 때 완성품처럼 보이는 3탭 구조를 만든다.
- 운동, 식단, 알람의 주요 화면 밀도를 확인한다.
- FAB와 bottom sheet 중심의 입력 흐름을 검증한다.
- 알람 울림 화면의 핵심 경험을 mock으로 먼저 체험한다.
- 이후 Cubit, DB, 알람 스케줄러, 센서 구현으로 자연스럽게 교체할 수 있는 화면 뼈대를 만든다.

---

## 전체 화면 구조

앱은 하단 탭 3개를 중심으로 구성한다.

| 탭 | 역할 | 핵심 액션 |
|---|---|---|
| 운동 | 날짜별 운동 기록 확인 | 운동 기록 추가 |
| 식단 | 날짜별 식단 기록 확인 | 식단 기록 추가 |
| 알람 | 미라클 모닝 알람 관리 | 알람 추가 |

mock UI에서는 실제 저장소를 사용하지 않고, 화면 안에 고정 mock 데이터를 배치한다.

---

## 1. 메인 앱 쉘

### 구성

- `MaterialApp`
- `Scaffold`
- `NavigationBar`
- 탭별 `FloatingActionButton`
- 탭별 body 전환

### 하단 탭

| 탭 이름 | 아이콘 예시 | 화면 |
|---|---|---|
| 운동 | `fitness_center_outlined` | `WorkoutMockTab` |
| 식단 | `restaurant_outlined` | `MealMockTab` |
| 알람 | `alarm_outlined` | `AlarmMockTab` |

### 기획 포인트

- 첫 화면은 `운동` 탭으로 시작한다.
- 탭 전환 시 화면 상태가 어색하게 초기화되지 않도록 추후 `IndexedStack` 사용을 고려한다.
- mock 단계에서는 화면별 Cubit 없이 `StatefulWidget` 안에서 간단한 상태만 관리한다.

---

## 2. 운동 탭

### 화면 목적

사용자가 오늘 또는 선택한 날짜의 운동 기록을 빠르게 확인한다.

### 구성

- 날짜 헤더
- 운동 기록 리스트
- 빈 상태
- 운동 추가 FAB

### 날짜 헤더

기존 `DateHeader` 컴포넌트를 사용한다.

표시 예시:

```text
2026년 5월 17일 일
```

액션:

- 이전 날짜
- 다음 날짜
- 달력 열기

mock 단계에서는 달력은 실제 구현하지 않고, 버튼 또는 bottom sheet placeholder로 처리할 수 있다.

### 운동 기록 카드

표시 정보:

- 기록 시간
- 운동 메모

예시 데이터:

| 시간 | 메모 |
|---|---|
| 07:20 | 상체 40분 / 푸시업 5세트 |
| 19:10 | 빠르게 걷기 30분 |

### 빈 상태

운동 기록이 없는 날짜에는 `EmptyState`를 사용한다.

문구:

```text
오늘 기록한 운동이 없어요
가볍게 한 줄만 남겨도 충분해요.
```

### 운동 기록 BottomSheet

기존 `AppBottomSheetScaffold`를 사용한다.

입력 요소:

- 메모 입력 필드
- 저장 버튼

mock 단계에서는 저장 버튼을 눌러도 실제 저장하지 않는다.
필요하면 화면 내 임시 리스트에만 추가한다.

---

## 3. 식단 탭

### 화면 목적

사용자가 날짜별 식단 기록을 사진과 함께 빠르게 확인한다.

### 구성

- 날짜 헤더
- 식단 기록 카드 리스트
- 빈 상태
- 식단 추가 FAB

### 식단 기록 카드

운동 기록보다 시각 정보를 더 강조한다.

표시 정보:

- 식사 사진 썸네일
- 기록 시간
- 식단 메모

예시 데이터:

| 시간 | 메모 | 사진 |
|---|---|---|
| 08:10 | 아침: 그릭요거트, 바나나 | placeholder |
| 12:40 | 점심: 닭가슴살 샐러드 | placeholder |
| 18:30 | 저녁: 현미밥, 계란, 김치 | placeholder |

### 사진 영역

mock 단계에서는 실제 `image_picker`를 연결하지 않는다.

대신 다음 중 하나를 사용한다.

- 색상 placeholder
- 음식 아이콘
- 앱 내부 mock asset

### 빈 상태

문구:

```text
오늘 기록한 식단이 없어요
사진 한 장이나 짧은 메모로 시작해보세요.
```

### 식단 기록 BottomSheet

입력 요소:

- 사진 추가 영역
- 카메라 버튼
- 갤러리 버튼
- 메모 입력 필드
- 저장 버튼

mock 단계에서는 카메라/갤러리 버튼을 눌렀을 때 실제 picker를 열지 않고, 선택된 것처럼 UI만 바꿔도 충분하다.

---

## 4. 알람 탭

### 화면 목적

사용자가 미라클 모닝 알람을 생성, 확인, 켜고 끌 수 있다.

### 구성

- 알람 목록
- 알람 On/Off switch
- 알람 추가 FAB
- 알람 울림 화면 mock 진입 경로

### 알람 카드

표시 정보:

- 큰 시간
- 라벨
- 반복 요일
- 흔들기 횟수
- On/Off switch

예시 데이터:

| 시간 | 라벨 | 반복 | 흔들기 |
|---|---|---|---|
| 05:30 | 미라클 모닝 | 월 화 수 목 금 | 20회 |
| 07:00 | 주말 산책 | 토 일 | 15회 |

### 카드 UI 예시

```text
05:30                 On
미라클 모닝
월 화 수 목 금 · 20회 흔들기
```

### 인터랙션

- switch: mock 상태만 변경
- 카드 탭: 알람 편집 bottom sheet 열기
- 길게 누름: 삭제 확인 dialog 연결 가능
- 별도 버튼: 알람 울림 화면 preview

### 기획 포인트

알람 탭은 앱의 차별점이 가장 잘 드러나는 영역이다.
mock 단계에서도 시간과 흔들기 횟수 정보가 한눈에 보여야 한다.

---

## 5. 알람 편집 BottomSheet

### 화면 목적

사용자가 알람의 시간, 반복 요일, 흔들기 횟수, 라벨을 설정한다.

### 구성

- 시간 선택 영역
- 요일 선택 chip
- 흔들기 횟수 설정
- 라벨 입력
- 저장 버튼

### 시간 선택

mock 단계에서는 실제 time picker를 연결하지 않아도 된다.

예시:

```text
05 : 30
```

추후 구현:

- `showTimePicker`
- 커스텀 wheel picker

### 요일 선택

요일 chip:

```text
월 화 수 목 금 토 일
```

선택 상태를 시각적으로 명확히 구분한다.

### 흔들기 횟수

범위:

```text
10회 ~ 50회
```

추천 UI:

- `Slider`
- 현재 값 텍스트

예시:

```text
20회 흔들어서 해제
```

### 라벨

기본값:

```text
미라클 모닝
```

## 6. 알람 울림 화면

### 화면 목적

알람이 울렸을 때 사용자가 반드시 일정 횟수 이상 흔들어야 해제되는 경험을 제공한다.

mock UI 단계에서 가장 중요한 검증 화면이다.

### 구성

- 전체 화면
- 큰 현재 시각
- 알람 라벨
- 남은 흔들기 안내
- 진행률 바
- 현재 카운트
- mock 흔들기 버튼

### 화면 예시

```text
05:30

미라클 모닝
20번 흔들어서 해제

[=========-----------]
7 / 20

흔들림 추가
```

### mock 동작

실제 `sensors_plus` 연결 전에는 버튼으로 흔들림을 시뮬레이션한다.

- 버튼 클릭 시 카운트 +1
- 목표 횟수 도달 시 화면 닫기 또는 완료 상태 표시
- 진행률 바 업데이트

### 추후 실제 구현 연결

- `AlarmRingingBloc`
- `ShakeDetector`
- `sensors_plus`
- 알람 사운드 정지
- 라우팅 pop

## 구현 순서 제안

1. `main.dart`를 3탭 mock 앱으로 변경한다.
2. 운동 탭을 mock 데이터로 만든다.
3. 식단 탭을 mock 데이터와 사진 placeholder로 만든다.
4. 알람 탭을 mock 데이터와 switch 상태로 만든다.
5. 각 탭 FAB에서 bottom sheet를 연다.
6. 알람 울림 mock 화면을 추가한다.
7. 작은 화면과 다크 모드에서 레이아웃을 확인한다.

---

## 화면별 성공 기준

### 운동 탭

- 날짜와 기록 목록이 한눈에 보인다.
- 기록이 없는 상태도 어색하지 않다.
- FAB를 눌렀을 때 입력 흐름이 자연스럽다.

### 식단 탭

- 사진 썸네일이 있어 식단 앱처럼 느껴진다.
- 텍스트와 사진의 밀도가 부담스럽지 않다.
- 사진 추가 흐름이 이후 실제 picker와 연결되기 쉽다.

### 알람 탭

- 시간, 반복, 흔들기 횟수가 즉시 이해된다.
- On/Off 상태가 명확하다.
- 알람 추가와 수정 흐름이 단순하다.

### 알람 울림 화면

- 사용자가 무엇을 해야 하는지 즉시 알 수 있다.
- 흔들기 진행률이 명확하다.
- MVP의 차별점이 화면만으로도 전달된다.

## 이후 실제 기능으로 교체할 부분

mock UI가 확정되면 다음 순서로 실제 기능을 붙인다.

1. 알람 POC
2. 알람 CRUD
3. 운동 DB + Cubit
4. 식단 DB + 사진 저장
5. 달력 네비게이션
6. 실기기 알람 검증
7. E2E 테스트

## 요약

mock UI의 핵심은 기능을 흉내 내는 것이 아니라 사용 흐름을 먼저 살아 있게 만드는 것이다.

첫 화면에서 사용자는 바로 날짜별 기록을 보고, FAB로 입력을 시작하고, 알람 탭에서 앱의 핵심 기능인 흔들기 해제 알람을 이해할 수 있어야 한다.

따라서 첫 mock 구현의 목표는 다음 한 문장으로 정리할 수 있다.

> 데이터는 가짜여도, 앱의 하루 사용 흐름은 진짜처럼 느껴지게 만든다.
